## G1 · Phase 0 想法圆桌

> 现状:Phase 0 圆桌机制已存在,本组主要是"沿用 + 衔接到新的关卡 0 + 方向落盘"。

### R-P0-1 三方质询打磨 idea
- **作用(WHY):** 把模糊 idea 逼问成清晰、有方向的问题,避免在错误前提上盖楼。
- **实现什么(WHAT):** arbiter 提出 idea;玄、素与 arbiter 三方反复质询、打磨,直到方向清晰。
- **怎么实现(HOW · 玄草案):** 沿用现有 Phase 0 圆桌(mailbox 交接 + channel 记录);玄偏综合收敛,素偏质疑发散。
- **验收:** channel.md 出现一段三方认可的"方向陈述"。
- **状态:** todo

### R-P0-2 方向落盘(files = memory)
- **作用(WHY):** 方向必须能被重启的 pane 重读,不靠会话记忆。
- **实现什么(WHAT):** 打磨结果(方向陈述)写入持久文件。
- **怎么实现(HOW · 玄草案):** 玄在收敛后,把"方向"小节写入 `_idea.md`(或约定的方向文件)。
- **验收:** `_idea.md` 含方向陈述;重启后 agent 读它即可恢复 Phase 0 结论。
- **状态:** todo

### R-P0-3 触发关卡 0
- **作用(WHY):** 需要一个明确的"Phase 0 结束"信号,而非含糊滑入下一阶段。
- **实现什么(WHAT):** 三方认为方向已清晰时,向 arbiter 发出"请确认方向"的交接。
- **怎么实现(HOW · 玄草案):** 玄写一条 `STATUS=gate` 的交接消息到 arbiter mailbox。
- **验收:** 出现一条指向 arbiter 的 gate 交接,arbiter 可据此 通过/打回。
- **状态:** todo

### R-P0-4 方向陈述达标(含 scope 边界)
- **作用(WHY):** 给关卡 0 一个可检验的"够清晰"标准;用"非目标/边界"为下游 ② 设防,防止 arbiter 抽身后架构膨胀。
- **实现什么(WHAT):** 方向陈述必须含 5 个最小要素——①核心问题 ②给谁用/价值 ③大致形态(一句话)④**明确不做什么(非目标)** ⑤已知关键未决问题;缺任一要素不得触发关卡 0(R-P0-3)。
- **怎么实现(HOW · 玄草案):** 玄按此 checklist 收敛方向陈述并自检;素对抗"边界是太宽还是太窄"。
- **验收:** `_idea.md` 方向陈述含全部 5 要素,且至少 1 条明确"非目标"。
- **状态:** todo

## G2 · 关卡 0 方向锁定

> 三道关卡中的第一道。关卡只有在"它批准的东西会被冻结"时才有意义——本组的重点在 R-G0-3。

### R-G0-1 自包含呈现
- **作用(WHY):** arbiter 要能"只读一份东西"就判断,不必翻聊天记录或多个文件。
- **实现什么(WHAT):** 触发关卡 0 时,完整呈现方向陈述(含 R-P0-4 的 5 要素),而非一句"我们觉得可以了"。
- **怎么实现(HOW · 玄草案):** 玄在 gate 交接消息里贴出方向陈述全文(或指向 `_idea.md` 的明确锚点)。
- **验收:** arbiter 不打开其它文件即可看到完整方向陈述。
- **状态:** todo

### R-G0-2 两种结果 + 落盘
- **作用(WHY):** 关卡结论必须持久、可追溯,重启的 pane 要能知道"方向已锁 / 仍在打磨"。
- **实现什么(WHAT):** arbiter 给出 **通过** 或 **打回(附理由)**;结论写入 `decisions.md`;打回则带理由退回 Phase 0 继续打磨。
- **怎么实现(HOW · 玄草案):** 玄收到 arbiter 裁决后,向 `decisions.md` 追加一行(Phase=0,Decided by=arbiter,含理由)。
- **验收:** `decisions.md` 出现该次关卡 0 的裁决记录;打回时 Phase 0 能据理由续做。
- **状态:** todo

### R-G0-3 通过即冻结(direction freeze)
- **作用(WHY):** 关卡只有"批准的东西会被冻结"才有意义——否则 ② 里 agent 又把方向推翻重来,失控且浪费。
- **实现什么(WHAT):** 关卡 0 通过后方向陈述标记为"已锁",成为 ② 的前提;② 中若发现方向必须变,**必须显式升级回 arbiter 重开关卡 0**,不得自行改方向。
- **怎么实现(HOW · 玄草案):** 在 `_idea.md` 方向小节加"🔒 已锁(关卡0 通过 @ 时间)"标记;协议规定改方向 = 回到关卡 0。
- **验收:** 方向陈述带"已锁"标记;② 阶段契约明确"改方向须重开关卡 0"。
- **状态:** todo

## G3 · Phase 1 最终形态设计

> 新流程的心脏:玄持笔产出"最终形态",素对抗评审。arbiter 只读(在关卡 1 介入)。
> panel 与关卡 1 分别在 G4 / G5。

### R-P1-1 产物构成
- **作用(WHY):** 明确 ② 的交付物到底是什么,否则"设计完成"无标准、无法过关卡 1。
- **实现什么(WHAT):** 玄持笔产出两份"最终形态"产物——`architecture.md`(整体架构)与 `flow.md`(业务逻辑流程,含必要的 Mermaid 图:至少流程图,按需补时序/状态机/数据流)。
- **怎么实现(HOW · 玄草案):** 复用 `templates/architecture.md` 与 `templates/flow.md` 骨架(见 G9),玄据冻结方向填充。
- **验收:** 产出 architecture.md 与 flow.md 两份文件,flow.md 至少含一张流程图。
- **状态:** todo

### R-P1-2 在冻结方向/边界内设计
- **作用(WHY):** arbiter 在 ② 抽身,已锁方向与"非目标"是防架构膨胀的护栏(呼应 R-P0-4 / R-G0-3)。
- **实现什么(WHAT):** 玄起草前先读 `_idea.md` 已锁方向;架构与流程不得突破"非目标";若边界确需改,升级回关卡 0,不在 ② 自行改方向。
- **怎么实现(HOW · 玄草案):** 玄在 architecture.md 顶部回链已锁方向;素负责挑"这里是不是越界了"。
- **验收:** architecture.md 显式引用已锁方向;无突破"非目标"的设计项。
- **状态:** todo

### R-P1-3 玄持笔 × 素对抗的回合
- **作用(WHY):** 设计质量来自对抗压力,不能玄一个人写完就算。
- **实现什么(WHAT):** 玄出初稿 → 素对抗评审(可行性/复杂度/边界/遗漏)并可贡献架构想法 → 玄修订;往复至两 agent 收敛。
- **怎么实现(HOW · 玄草案):** 沿用 mailbox 交接 + channel 记录;每轮素的评审落到 channel,玄据此更新产物。
- **验收:** channel 中可见至少一轮"素评审 → 玄修订";收敛时两 agent 均认可。
- **状态:** todo

### R-P1-4 生产运行时视角
- **作用(WHY):** arbiter 明确要"生产环境运行时的业务逻辑流程";它是 ③ 拆需求的骨架,画成开发流程就跑偏。
- **实现什么(WHAT):** flow.md 以"系统在生产环境运行时的业务逻辑流转"为视角,**节点 = 业务步骤**(这些节点正是 ③ 要逐个展开成需求的对象)。
- **怎么实现(HOW · 玄草案):** flow.md 用 Mermaid 画运行时业务流;玄确保节点是业务步骤,而非"先写A再写B"的开发步骤。
- **验收:** flow.md 节点为业务步骤,可直接作为 ③ 的拆解清单来源。
- **状态:** todo

### R-P1-5 产物落盘且自包含
- **作用(WHY):** files=memory;关卡 1 你只读这两份就能判断;重启 pane 重读即可恢复 ②。
- **实现什么(WHAT):** architecture.md / flow.md 写入约定路径,内容自包含,不依赖会话上下文。
- **怎么实现(HOW · 玄草案):** 写入项目内约定 design 目录(具体路径在 G9 定);玄自检"脱离聊天也能读懂"。
- **验收:** 新开 pane 仅凭这两份文件即可理解最终形态。
- **状态:** todo

### R-P1-6 收敛信号
- **作用(WHY):** 让 ② 的"出口"显式化(与 R-P0-3 对称),重启 pane 一眼看出 ② 走到哪;为 panel/关卡 1 提供明确触发点。
- **实现什么(WHAT):** 两 agent 对最终形态达成一致后,玄标记"准最终稿(待 panel)",作为 ② 的收敛信号,触发 G4 panel;panel 补完后再触发关卡 1。
- **怎么实现(HOW · 玄草案):** 玄在产物顶部/channel 写 `STATUS=converged` 标记,随后按 G4 派生 panel。
- **验收:** 出现明确"准最终稿待 panel"信号,且此前两 agent 均认可。
- **状态:** todo

## G4 · 专项评审 panel 机制

> panel = 持笔方在"收敛后、交 arbiter 前"派生的一次性专项评审子 agent。② 强制、③ 按需。
> 设计红线:守住"极小中继 / pane 一次性",**不增常驻 agent、不碰 relay**。

### R-P4-1 触发与位置(②强制 / ③按需)
- **作用(WHY):** panel 必须跑在"两 agent 收敛后、交 arbiter 前",才能在你看到前扫掉盲点;② 影响最大故强制,③ 强制会拖垮"逐条"节奏故按需。
- **实现什么(WHAT):** ② 收到收敛信号(R-P1-6)后**强制**触发 panel;③ 中某条需求满足"高复杂/高风险/跨多节点"判据时才触发,否则跳过;panel 一律在关卡前完成。
- **怎么实现(HOW · 玄草案):** 玄收到收敛信号后派生一组专项评审子 agent;③ 中按判据为需要的需求项标注 `needs-panel`。
- **验收:** ② 每次过关卡前必有 panel 记录;③ 有明确判据决定是否跑,且记录跑/跳过。
- **状态:** todo

### R-P4-2 一次性 · 不入圆桌 · 不碰 relay
- **作用(WHY):** 守住"极小中继 / pane 一次性";不能因评审而增加常驻 agent 或 relay 路由。
- **实现什么(WHAT):** panel 子 agent 由持笔方内部派生,跑完即销毁;人类可见圆桌仍是 2 个 pane;不经 relay/mailbox。
- **怎么实现(HOW · 玄草案):** 用持笔 agent 自身的 subagent 能力(如 Claude Code 的 Task/Agent),而非新开 tmux pane。
- **验收:** panel 期间 tmux pane 数不变;relay 路由不涉及 panel。
- **状态:** todo

### R-P4-3 评审视角(lens)集
- **作用(WHY):** 多视角覆盖盲点,但要可控,不是无限加 agent。
- **实现什么(WHAT):** 定义一组标准 lens——可行性 / 安全 / 范围蔓延 / 数据流 / 运维可靠性……;持笔方按产物性质**选用相关的若干**,而非全开。
- **怎么实现(HOW · 玄草案):** 每个选中的 lens 一个子 agent,各审一遍同一份产物。
- **验收:** panel 记录中可见所用 lens 清单及各自发现。
- **状态:** todo

### R-P4-4 发现回流与归并
- **作用(WHY):** panel 的价值在于发现被持笔方吸收,而不是堆一堆没人管的意见。
- **实现什么(WHAT):** 各 lens 交回发现;持笔方(玄)逐条决定采纳/拒绝并说明,更新产物;重大分歧或被拒的重要意见落 `decisions.md`,随后触发关卡。
- **怎么实现(HOW · 玄草案):** 玄汇总 panel 发现 → 逐条处置 → 更新 architecture/flow(或 requirements)→ 触发关卡。
- **验收:** 每条 panel 发现都有"采纳/拒绝 + 理由";产物或 decisions.md 体现处置结果。
- **状态:** todo

## G5 · 关卡 1 最终形态确认

> 三道关卡的第二道,也是 arbiter 在 ② 唯一的介入点。结构与 G2(关卡 0)平行,
> 区别:打回**可一路退到关卡 0 换大方向**。

### R-G1-1 只读可判
- **作用(WHY):** arbiter 在 ② 抽身,关卡 1 是唯一介入点;必须能"只读产物"就判断方向对不对,不必翻 channel。
- **实现什么(WHAT):** 触发关卡 1 时呈现:最终形态两份产物(architecture.md + flow.md,已含 panel 处置)+ 一份 panel 发现处置小结;自包含。
- **怎么实现(HOW · 玄草案):** 玄在 gate 交接里给出产物锚点 + panel 小结,确保 arbiter 不需读 channel。
- **验收:** arbiter 仅凭产物 + panel 小结即可判断,无需翻聊天/channel。
- **状态:** todo

### R-G1-2 两种结果 + 落盘
- **作用(WHY):** 结论须持久可追溯;打回要能带理由回到正确位置。
- **实现什么(WHAT):** arbiter 给 **通过** / **打回(附理由)**;打回可指明"回 ② 改产物"或"方向本身要变 → 退回关卡 0";结论写入 `decisions.md`;通过则进入 ③。
- **怎么实现(HOW · 玄草案):** 玄据裁决向 decisions.md 追加一行(Phase=1,Decided by=arbiter,含理由与去向)。
- **验收:** decisions.md 出现该次关卡 1 裁决;打回时能据理由回到 ② 或关卡 0。
- **状态:** todo

### R-G1-3 通过即冻结最终形态
- **作用(WHY):** 同 R-G0-3 的粘性逻辑——③ 拆需求必须基于稳定的最终形态;否则 ③ 边拆边变,失控。
- **实现什么(WHAT):** 通过后 architecture/flow 标"已锁",成为 ③ 的前提;③ 中若发现最终形态有硬伤,**显式升级回关卡 1 重审**,不在 ③ 自行改架构/流程。
- **怎么实现(HOW · 玄草案):** 产物顶部加"🔒 已锁(关卡1 通过 @ 时间)";协议规定改最终形态 = 回关卡 1。
- **验收:** 产物带"已锁"标记;③ 契约明确"改最终形态须重开关卡 1"。
- **状态:** todo

## G6 · Phase 2 详细需求

> 自指组:把"arbiter 参与每一条需求"的机制本身写成需求。这是本 idea 的核心。
> (我们当前正用这套机制走 G1...G12,即是它的活样板。)

### R-P2-1 拆解来源 = flow 节点
- **作用(WHY):** 保证需求覆盖全部业务逻辑,且每条可追溯到一个流程节点(粒度锚点)。
- **实现什么(WHAT):** 以已锁 `flow.md` 的每个业务步骤节点为拆解清单,逐节点展开成多条原子需求(n:1);每条记 `所属节点`。
- **怎么实现(HOW · 玄草案):** 玄把 flow 节点列成清单,逐节点起草需求。
- **验收:** 每个 flow 节点都被展开;每条需求都标了所属节点。
- **状态:** todo

### R-P2-2 原子需求字段完备
- **作用(WHY):** "每一条都清晰明了"——缺字段就不算讲清楚,构建期无法据其落地。
- **实现什么(WHAT):** 每条需求含 `编号/所属节点/作用(WHY)/实现什么(WHAT)/怎么实现(HOW)/验收/状态`;缺字段不得进关卡 2。
- **怎么实现(HOW · 玄草案):** requirements 模板(G9)强约束字段;玄自检完备性。
- **验收:** 抽查任一需求,七字段齐全。
- **状态:** todo

### R-P2-3 按字段分工
- **作用(WHY):** 你要参与"做什么、有什么用",但"怎么做"交执行方,减负且分工清晰。
- **实现什么(WHAT):** 玄逐条草拟 WHY/WHAT → 你和素挑战 → **你拍板 WHY/WHAT**;HOW 由玄/素定,你可不介入。
- **怎么实现(HOW · 玄草案):** 交接时玄把每条的 WHY/WHAT 高亮给 arbiter,HOW 折叠待 agent 内部敲定。
- **验收:** 每条需求的 WHY/WHAT 有 arbiter 痕迹;HOW 由 agent 完成。
- **状态:** todo

### R-P2-4 你参与每一条,无一跳过(覆盖=100%)
- **作用(WHY):** 你明确要求参与每个需求;不能抽样或批量略过。
- **实现什么(WHAT):** 每条需求的 WHY/WHAT 必须经你确认或修改才能 lock;未确认者标 `pending`,**不得进关卡 2**。
- **怎么实现(HOW · 玄草案):** 两 agent 先备好每条 WHY/WHAT 草案,你逐条确认;流程上以 pending 计数卡住关卡 2。
